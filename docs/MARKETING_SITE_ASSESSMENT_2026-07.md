# Marketing Site Assessment — Averrow (2026-07)

**Scope:** audit of `packages/averrow-marketing` (the public site served at
`averrow.com`) for **accuracy, content, UI/design, and code/SEO**; a
**competitor benchmark** against best-in-class security peers; a **restructure
blueprint** splitting the site into a **Product** side and a **Company** side on
one domain; a **gap analysis** vs. a full-parity target; and a **content-agent
plan** to produce the missing pages in parallel.

**Bottom line:** the site is well-built but **thin**. Code quality is genuinely
good (clean TypeScript, solid responsive, accessible forms). What holds it back
is **breadth, proof, and discoverability** — the layers a security buyer reads
as "this is a real, operating company." Closing that gap is an **additive
content + IA project**, not a redesign. Several **accuracy and positioning
defects** should be fixed first because they actively undercut credibility.

> This is a strategy + audit deliverable. It changes no site code. The companion
> work in this pass is the four new content sub-agents (see §7 and
> `docs/CLAUDE_SUBAGENTS.md`). Building the pages is the next pass, gated on review.

---

## 1. Executive summary — does it compare, and how do we significantly lift it?

**How it compares today.** Peers (Recorded Future, ZeroFox, Bolster, Doppel,
Netcraft, BrandShield, Corsearch) all converge on the same information
architecture: **Platform/Product · Solutions · Customers · Resources · Pricing ·
Company (About/Careers/Press/Partners/Trust/Legal) · utility (Login / Get a
Demo)**. Averrow has the **spine** of that (home, platform, pricing, about,
security, blog, changelog, contact) but is **missing the credibility and depth
layers**: solutions-by-persona, customer proof/case studies, product deep-dives,
resources/research, docs, partners, and a real corporate/trust surface. It reads
as an early product site, not an operating security company.

**The three things that most limit the site right now:**
1. **No proof.** The only social proof is a *disclosed placeholder* testimonial
   and "customer logos publishing in 2026." Security buyers treat proof as
   table stakes; its absence reads as pre-revenue.
2. **No depth for the evaluator.** A CISO or SOC lead has nowhere to go deep —
   no product deep-dives, no solutions for their persona, no docs. Peers give
   each capability its own page.
3. **Weak discoverability.** No Open Graph / Twitter / JSON-LD means every shared
   link renders blank and search engines get no structured signal — despite an
   OG image asset already existing at `/brand/averrow-og.png`.

**How to significantly lift it (in priority order):**
- **P1 — Fix credibility defects (days).** Reconcile the contradictory
  agent/feed numbers, remove Canada-first cues, replace placeholder proof, wire
  SEO/OG tags. These are small edits with outsized trust impact.
- **P2 — Add proof + persona depth (weeks).** Customers/case-studies surface,
  solutions-by-persona pages, product deep-dives. This is where conversion moves.
- **P3 — Add the resource + reference layer.** Resources/research hub, docs hub,
  partners/MSSP program. This is what makes the company look established and
  feeds SEO.
- **P4 — Build out the corporate surface.** A real Company section (leadership,
  careers, press, trust center, DPA) reached from the footer.

This mirrors the archived `docs/archive/CORPORATE_SITE_PLAN.md`, which specced
the full site but was only partially built.

## 2. Current state (verified)

- **Stack:** Astro 4.16 static SSG. Plain CSS custom-property tokens
  (`src/styles/tokens.css` + `global.css`), no Tailwind. One React island
  (`ThemeCycle.tsx`). MDX blog + RSS feeds.
- **Serving:** built to `dist/`, then `scripts/sync-to-worker.mjs` overlay-copies
  into `packages/averrow-worker/public/`; the averrow Worker serves it via the
  Cloudflare `ASSETS` binding. Canonical domain `averrow.com` (all others 301 to
  it). Root `/` = marketing home. App lives at `/v2/*` (staff) and `/tenant/*`.
- **Pages today (Astro):** `/`, `/platform`, `/pricing`, `/about`, `/security`,
  `/contact`, `/abuse-mailbox`, `/report-abuse`, `/blog` (+`[...slug]`, feed),
  `/changelog` (+feed).
- **Worker-rendered (not Astro):** `/privacy`, `/terms`, `/team`, `/assess`,
  `/status`, `/legacy`. Sitemap + `robots.txt` are also generated by the Worker,
  not this package.
- **Design:** intentional token parity with the app (ported from
  `packages/averrow-worker/src/templates/shared.ts`). Dark default + light theme.

There is **no Product-vs-Company split inside marketing today** — the only split
is marketing (`/`) vs. app (`/v2`, `/tenant`).

## 3. Audit findings

Severity: **P1** = actively undercuts credibility, fix first · **P2** = notable ·
**P3** = polish. Every finding was re-checked against the live file.

### 3.1 Accuracy (P1 — internal contradictions a prospect will notice)

| # | Finding | Evidence | Fix |
|---|---|---|---|
| A1 | **Agent count contradicts itself: 18 vs 27.** Home says "Six core agents. **18** working continuously"; Platform hero says "**Twenty-seven** agents." | `index.astro:142` (via `stats.json` `agents_deployed:"18"`) vs `platform.astro:20,15` | Pick one true number; make it the single source (ideally driven from `stats.json`, which the Platform page currently ignores). |
| A2 | **Feed count disagrees three ways: 33+ / 45+ / 6+.** | `stats.json:3` `"33+"`; `pricing.astro:56` "45+ feeds… by six AI agents"; `about.astro:120` "6+" | Reconcile to one verified number; distinguish "feeds" from "integrated brand-protection feeds" if they mean different things. |
| A3 | **Cost claim stated two ways on the same About page:** "10–50× lower" vs "50–66% less." | `about.astro` (story/principles vs Company Facts tile) | Choose one framing and one figure; back it with a source note. |
| A4 | **`stats.json` is a static fallback dated 2026-01-01.** Homepage stat counters (`210K+` threats, `9.6K+` brands) are hardcoded placeholders when the build-time fetch fails. | `stats.json:8-9` (`"source":"static-fallback"`) | Confirm `scripts/fetch-stats.mjs` runs against prod at build; refresh the committed fallback so a fetch failure doesn't ship year-old numbers. |

### 3.2 Positioning / brand compliance (P1 — violates CLAUDE.md §13 "global, not Canada-first")

| # | Finding | Evidence | Fix |
|---|---|---|---|
| P1 | **Canada-first cue on Contact:** a 🇨🇦 flag emoji + "LRX Enterprises Inc., **Canada**." Most Canada-forward element on the site. | `contact.astro:107` (`&#127464;&#127462;`), `:110` | Drop the flag; use a neutral "LRX Enterprises Inc." (legal entity can stay; the *flag + country-forward* framing is the issue). |
| P2 | **PIPEDA listed before GDPR** in the compliance timeline — Canada-first ordering/emphasis. | `security.astro:179` (PIPEDA) precedes `:184` (GDPR) | Lead with GDPR (or present both neutrally); PIPEDA can remain listed. |
| P3 | **Abuse emails on `.ca` while the rest of the site is `.com`.** Reads Canada-leaning and inconsistent. | `report-abuse.astro:49,77,104` (`phishing@averrow.ca`, `abuse@averrow.ca`) vs `.com` on contact/security | Unify to `.com` (or confirm the `.ca` mailboxes are intentional and consistent everywhere). |
| P4 | **Aviation remnants (cosmetic, low severity):** internal CSS class names `squadron*` and a decorative delta-"wing" SVG on About. **Not user-visible text.** | `index.astro:140,142,746+`; `platform.astro:26-37,278+`; `about.astro:18,34` (`about-hero-wing`, `about-heritage`) | Optional cleanup. **Note:** the Avro Arrow *logo mark* is explicitly sanctioned by `BRAND.md` as an About-page heritage nod — leave the mark; only the `squadron` class names / wing echo are stylistic leftovers. |

### 3.3 Content (P2 — placeholders and gaps)

- **C1 (P1-adjacent): No real social proof.** `TrustSignals.astro` ships generic
  sector icons with a disclosed "Sample placement — customer logos publishing in
  2026" and a **fabricated testimonial** ("VP of Security · Mid-market SaaS · 800
  employees"). Replace with real (even if anonymized-with-permission) proof, or
  remove until available — a disclosed-fake testimonial is worse than none.
- **C2: Single blog byline "Claude Leroux"** across all five posts reads as a
  placeholder / AI pun. Confirm the intended author identity/policy.
- **C3: Thin blog (5 posts, 1 draft).** A security company signals life through
  regular research/threat content. This is the top-of-funnel SEO engine and it's
  nearly empty.
- **C4: Missing entire content surfaces** — see the gap table in §5.

### 3.4 UI / design (P2/P3 — mostly good, targeted issues)

- Strengths: consistent token system, dark/light parity, `clamp()` fluid type,
  grids that collapse cleanly, `prefers-reduced-motion` honored, dedicated mobile
  menu. This is a solid visual foundation to build on.
- **U1 (a11y, P2): No `<main>` landmark and no skip link.** Primary content is a
  plain `<div class="page-content">` on every page. `Layout.astro:71`. Add
  `<main id="content">` + a skip link.
- **U2 (a11y, P2): The homepage scan input — the site's primary CTA — has only a
  placeholder, no label.** `index.astro` scan form. Add a visually-hidden
  `<label>` or `aria-label`.
- **U3 (P3): Logo SVG is copy-pasted** in `Nav.astro` and `Footer.astro` (only
  the gradient id differs). Extract a shared `Logo.astro`.
- **U4 (P3): Reuse nits** — `base` replace repeated per-file; badge CSS
  duplicated in `blog/[...slug].astro`.

### 3.5 Code / SEO / performance (P1 for SEO — the single weakest area)

- **S1 (P1): No Open Graph, Twitter Card, or JSON-LD anywhere.** Shared links
  render blank; search engines get no structured data. `Layout.astro` head
  (`:30-44`) sets title/description/canonical/RSS but nothing social/structured.
  **The OG image already exists** at `/brand/averrow-og.png` (per `BRAND.md`) —
  it just isn't referenced. **Highest-ROI fix on the site:** one edit to
  `Layout.astro` adds `og:*` + `twitter:*` + an `Organization` JSON-LD block to
  every page; add `Article` JSON-LD to blog posts (frontmatter already carries
  author/date/excerpt).
- **S2 (P2): Sitemap + robots.txt are generated by the Worker, not this
  package** (`@astrojs/sitemap` deferred due to an Astro 4.16 build crash). This
  couples crawler infra to the Worker and risks drift when routes change.
  Re-enable native sitemap generation or add a check that the Worker handler
  stays in sync.
- **S3 (P2, perf): Render-blocking Google Fonts** — 10 weights across two
  families via a blocking `<link>` with no `preconnect`/self-host.
  `Layout.astro:41-44`. Likely the main LCP/CLS drag. Subset the weights,
  `preconnect`, or self-host.
- **S4 (P3): `ThemeCycle` hydrates `client:load`** though it's a non-critical
  toggle (the pre-paint script already prevents flash). `client:idle` would defer it.
- **S5 (P3, consistency): `changelog-entries.ts` is duplicated** between this
  package and `packages/averrow-worker/src/templates/` and kept in manual sync — a
  drift risk. Consider a single shared source.

## 4. Competitor benchmark

Peer sites (Recorded Future, ZeroFox, Bolster, Doppel, Netcraft) block automated
fetching, so this synthesizes their well-known IA + 2026 B2B-security web
best-practice research. The **shared pattern** every serious peer follows:

- **Homepage** answers *what problem / what product / who for* within ~5s, then
  proof, then a single clear CTA.
- **A page per capability** (product deep-dives) for the technical evaluator.
- **Solutions by persona/segment** (CISO vs SOC vs mid-market vs MSSP).
- **Customers/case studies with hard numbers** — proof is everywhere,
  contextually (logos on home, cases on product, badges on pricing).
- **A resources/research hub** — threat reports, guides, webinars, glossary —
  the SEO + thought-leadership engine.
- **Docs** — signals a real, integrable product.
- **A trust/security center + full Company section** — About, leadership,
  careers, press, partners, legal/DPA.
- **Utility:** persistent "Get a Demo" primary CTA + Login.

### Scorecard — Averrow vs. peer-standard IA

| Section | Peer standard | Averrow today | Gap |
|---|---|---|---|
| Homepage clarity | ✅ | ✅ strong hero, clear value prop | Minor |
| Product overview | ✅ | ✅ `/platform` | Minor |
| Product deep-dives | ✅ page each | ❌ none | **Large** |
| Solutions by persona | ✅ | ❌ none | **Large** |
| Customers / case studies | ✅ | ❌ placeholder only | **Large** |
| Pricing | ✅ | ✅ `/pricing` (accurate tiers) | None |
| Resources / research | ✅ hub | 🟡 blog only (5 posts) | **Large** |
| Docs | ✅ | ❌ none | **Large** |
| Partners / MSSP | ✅ | ❌ none | Medium |
| Company / About | ✅ + team | 🟡 `/about`, no team/leadership | Medium |
| Careers | ✅ | ❌ none | Medium |
| Press / newsroom | ✅ | ❌ none | Medium |
| Trust / security center | ✅ | ✅ `/security` (good) | Minor |
| Legal (Privacy/Terms/DPA) | ✅ | 🟡 Privacy/Terms yes, **no DPA** | Medium |
| SEO / shareability | ✅ | ❌ no OG/JSON-LD | **Large** |
| Primary CTA (Get a Demo) | ✅ persistent | 🟡 Free Scan present; no demo path | Medium |

**Verdict:** Averrow is competitive on the *foundation* (homepage, platform,
pricing, trust page, visual quality) and behind on the *depth + proof + reach*
layers. Parity is achievable additively.

## 6. Restructure blueprint — Product side + Company side, one domain

The user's model: the **root URL lands on the Product side**; the **footer is the
gateway to the Company/Corporate side**; sub-pages branch from each; everything
on one domain (`averrow.com`). Recommended realization:

### Header nav = Product-led (the buyer journey)
```
Platform ▾            Solutions ▾           Customers   Resources ▾      Pricing
  Overview              Mid-market            (proof)     Blog
  Threat Detection      MSSP                              Research/Reports
  Email Security        Startups                          Docs
  Social Monitoring                                       Changelog
  AI Agents
                                          [ Log in ]   [ Get a Demo ]  ← persistent primary CTA
```
- Primary CTA is **Get a Demo**; **Free Scan** remains the secondary,
  low-friction entry (it's a genuine differentiator — keep it prominent on the
  homepage hero).

### Footer = the gateway to the Company/Corporate side
Recommended label: **"Company"** (clearer and more conventional than
"Corporate"; peers use "Company"). Add a **`/company` hub page** that links the
corporate cluster, and surface these columns in the footer:
```
Product            Solutions         Company              Resources          Legal
  Platform           Mid-market        About                Blog               Privacy
  Deep-dives         MSSP              Leadership           Research           Terms
  Pricing            Startups          Careers              Docs               DPA
  Free Scan          Customers         Press                Changelog          Security/Trust
                                       Partners             Report Abuse
                                       Contact
```

### Why path-based on one domain (not a subdomain)
- Keeps link equity + brand consolidated on `averrow.com` (SEO + trust).
- Fits the existing model perfectly: Astro file-routing produces
  `/company/index.html`, `/solutions/mssp/index.html`, etc., and the Worker's
  `ASSETS` binding serves them with zero routing changes.
- The corporate pages live at clean top-level/`/company/*` paths but are
  **presented as the Company side via the footer + `/company` hub** — exactly the
  "product up top, company in the footer" model requested.

### Implementation notes for the build pass
- Reuse `Layout.astro` for all new pages; add nav/footer entries in
  `src/lib/nav.ts` + `components/Footer.astro` (both are already data-driven).
- Extract shared marketing primitives before mass page creation: `Logo.astro`,
  `Hero.astro`, `FeatureRow.astro`, `CTASection.astro`, `Card.astro` — so 20+ new
  pages compose instead of copy-paste (the `CORPORATE_SITE_PLAN` "Shared
  Components" list is a good starting inventory).
- Every new route must be added to the sitemap generator (§3.5 S2) and, if
  Worker-served today (`/privacy`,`/terms`), decide migrate-vs-keep.
- Do **not** touch `public/`, `app.js`, `styles.css`, or frozen components.

## 7. Content-production plan — new sub-agents + parallel workflow

Doc/page writing is the slow part; the user asked to parallelize it. The current
roster has **one** content writer (`content-strategist`) and **no** agent for
long-form pages, competitive research, SEO, or legal copy. This pass adds four
specialists (definitions in `.claude/agents/`, registered in
`docs/CLAUDE_SUBAGENTS.md`):

| New agent | Model | Owns |
|---|---|---|
| **`web-copywriter`** | sonnet | Long-form marketing + company page copy, thought-leadership, blog. **The fan-out workhorse** — run N instances, one per page. |
| **`market-analyst`** | sonnet | Competitive/market intelligence: peer teardowns, positioning/messaging analysis, category research. Feeds briefs + this assessment. |
| **`seo-strategist`** | sonnet | Technical + content SEO: OG/Twitter/JSON-LD, sitemap/robots, keyword map, internal linking, meta. Owns the S1/S2 gaps. |
| **`legal-content-drafter`** | sonnet | Drafts Privacy/Terms/DPA/trust-center copy — every output stamped "DRAFT — requires human/legal review, not legal advice." |

**Existing agents keep their lanes:** `content-strategist` = editor-in-chief
(brand-voice gate, writes the per-page briefs); `design-reviewer` +
`seo-strategist` review; `frontend-engineer` builds the Astro pages;
`docs-maintainer` keeps internal docs synced; `qa-verifier` runs the gate.

### Parallel-production pattern (for the build pass — documented, not run here)
```
content-strategist writes per-page briefs (voice + facts + structure)
        │
        ├─ fan out: web-copywriter × N   (one per page, IN PARALLEL)
        │      → market-analyst supplies competitive framing per brief
        │
        ├─ per page: seo-strategist (meta/OG/JSON-LD/keywords)
        │            design-reviewer (layout/token/a11y)
        │
        └─ frontend-engineer builds the Astro page → qa-verifier gate
```
For a batch of 15–20 pages this is a natural **fan-out**: launch multiple
`web-copywriter` agents in one message (or a Workflow, with explicit user
opt-in given the token cost). Dedup guardrails: every writer gets the same
brand-voice brief + the §3 defect list so no one re-introduces Canada-first
framing or contradictory numbers.
